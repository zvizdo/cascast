# CLAUDE.md

Behavioral guidelines to reduce common LLM coding mistakes. Merge with project-specific instructions as needed.

**Tradeoff:** These guidelines bias toward caution over speed. For trivial tasks, use judgment.

## 1. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them - don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

## 2. Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

## 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

When editing existing code:
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it - don't delete it.

When your changes create orphans:
- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.

The test: Every changed line should trace directly to the user's request.

## 4. Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:
```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.

---

**These guidelines are working if:** fewer unnecessary changes in diffs, fewer rewrites due to overcomplication, and clarifying questions come before implementation rather than after mistakes.

## Learnings / gotchas (carry forward)
- **An R3F (`@react-three/fiber`) `<Canvas>` can only contain THREE objects — a plain DOM `<div>` (e.g. a Suspense/ErrorBoundary fallback) placed INSIDE `<Canvas>` throws `R3F: Div is not part of the THREE namespace`.** Visible DOM fallbacks for 3D failure (WebGL unsupported / context-lost / GLB load error) must live OUTSIDE the `<Canvas>` (an outer error boundary), or be wrapped in drei `<Html>`. The inner `<Suspense>`/boundary fallbacks stay R3F-safe (`null` or a 3D object). `Mountain3D` uses an outer `GLErrorBoundary` for the DOM message and `null` inner fallbacks. Caught only at runtime (jsdom + the route-mocked e2e don't mount WebGL); verify on a real WebGL browser / live deploy.
- **Headless Playwright (incl. the MCP browser) has unreliable WebGL** — `THREE.WebGLRenderer: Context Lost` is common, so the /3d model often won't render there even though it works on real devices (data endpoints are 200). Don't conclude /3d is broken from a headless "3D model not available" — check the terrain `meta`/`model` routes return 200 and verify on a real GPU browser. `src/components/three/**` is coverage-excluded for the same reason.
- **3D terrain looked like a "dark blob" = trimesh's glTF material has `metalness=1`** (the glTF default), which renders near-black without an environment map and hides the baked vertex colors. The viewer replaces each terrain mesh's material with a matte `MeshStandardMaterial({ vertexColors, metalness:0, roughness:0.95 })` on load. If you re-introduce the GLB material, the relief disappears.
- **Terrain GLB/meta are NOT `immutable`-cached** (`max-age=3600, stale-while-revalidate=86400`) so a re-bake (palette/geometry) propagates to clients within the hour. Don't restore `immutable` — re-bakes would pin stale terrain forever.
- **Local 3D visual iteration:** `npx next dev --webpack` (Turbopack `next dev` panics on globals.css) + a `.env.local` (gitignored) with `GCP_PROJECT`/`GCS_BUCKET_*` so the API routes read the real terrain/satellite GCS objects via ADC. Re-baking writes to the single prod bucket, so local dev sees new bakes immediately. Bust the browser's terrain cache with a fresh Playwright context.
- **trimesh GLB has no usable vertex normals** with `process=False`; any shader needing normals must derive them (e.g. `cross(dFdx(worldPos), dFdy(worldPos))`), not read the `normal` attribute. This is why SlopeShading uses a derivative normal.
- **The terrain bake CLI is a one-shot, run from REPO ROOT with `PYTHONPATH=functions`.** Two-step: `npx tsx scripts/export-peaks.ts` (writes the gitignored `functions/tools/peaks.json` from the TS catalog) then `PYTHONPATH=functions GCS_BUCKET_TERRAIN=mountain-weatherman-app-terrain GCP_PROJECT=mountain-weatherman-app functions/.venv/bin/python -m functions.tools.build_terrain --mountain <slug>` (or `--all`). `PYTHONPATH=functions` is required because the vendored `storage_client` does `from shared import config`. Needs `functions/.venv` with `requirements-terrain.txt` (rasterio/numpy/trimesh) installed (`uv pip install`, the venv is uv-managed). The bake is idempotent (re-running overwrites). **Adding a mountain now also needs a terrain bake + a `public/routes/<slug>.geojson`** (see references/add-mountain.md).
- **`.geojson` is NOT importable under Turbopack** (Next 16 `next build` uses Turbopack, which treats unknown extensions as asset URLs, not parsed JSON). Serve route data from `public/routes/` and `fetch` it; never `import x from "*.geojson"`.
- **The terrain pytest suite must run from the repo root** (`functions/.venv/bin/python -m pytest functions/tools/tests/ --rootdir=. -p no:cov -o addopts=""`) because it imports `functions.tools.*`; the default `functions/`-rooted gate ignores `tools/` (pyproject `--ignore=tools` + coverage `omit`). Run BOTH at gate time.
- **Adding a mountain needs BOTH a Firestore seed AND a web redeploy.** The web API serves the catalog from the bundled `src/lib/mountains-data.ts` constant (not Firestore), while the Python functions read the catalog from Firestore. They're seeded from the same file but are separate sources — update the constant, `npm run seed:mountains`, then `terraform apply` (web image rebuild). The mountain detail/Model-Lab pages also read the constant, so without the redeploy a new slug 404s in the UI.
- **`Firestore.doc("")` throws.** Any route or worker that builds a doc ref from a mountain field (`nwacZoneId`, `snotelStationTriplet`) must guard the empty case BEFORE constructing the ref. Out-of-region peaks (e.g. Mount Whitney) carry empty strings for the fields they lack.
- **Out-of-region peaks are supported via empty strings + worker skips.** NWAC covers WA+OR only and NRCS SNOTEL is sparse outside the Cascades. For such peaks set `nwacZone*`/`snotelStation*` to `""`; the snotel/nwac workers early-return on empty station/zone, the orchestrator still fans out to them (it doesn't filter), and the UI/API degrade to off-season/pending. Weather + satellite are global and still populate.
- **`src/lib/__tests__/mountains-data.test.ts` encodes WA/OR assumptions** — coordinate bounds, the `:(WA|OR):SNTL` triplet regex, and NWAC_ZONE_IDS membership. Adding an out-of-region peak requires widening the coord box and making the zone/station checks conditional (`if (m.nwacZoneId) ...`).
- **Deploy = plan-then-apply, never blind `-auto-approve`** (the deploy guard blocks it on the single prod env). Run `terraform -chdir=terraform plan -out=PLAN` then `terraform -chdir=terraform apply PLAN`. A normal redeploy shows ~7 add / 7 change / 7 destroy = hash-triggered restage of the web image + the 5 function source zips (replacements) + budget tweak — NOT destructive (no Firestore/bucket loss).
- **Deploy needs BOTH `TF_VAR_alert_email` AND `TF_VAR_ga_measurement_id` exported, or the apply silently wipes live config.** Both are non-committed-by-convention vars that default to `""`. Without `TF_VAR_alert_email` the monitoring email channel is deleted; without `TF_VAR_ga_measurement_id` the Cloud Run `GA_MEASUREMENT_ID` env (live value `G-P3C545NTG6`) is removed, killing site analytics. ALWAYS deploy with `TF_VAR_alert_email="<email>" TF_VAR_ga_measurement_id="G-P3C545NTG6" terraform -chdir=terraform plan/apply`, and check the plan for a `module.web…GA_MEASUREMENT_ID … -> null` or `module.monitoring…` channel deletion — if you see either, the env var is missing.
- **Open-Meteo throttling at fan-out scale = retry the TRANSIENT 400/429, jitter the herd.** The orchestrator fans the weather refresh out to ALL mountains in the same minute and each `weather-worker` makes 2 concurrent Open-Meteo calls, so a large catalog (50+) bursts the public API → HTTP 400 `{reason:"Too many concurrent requests"}` / HTTP 429. Open-Meteo signals the concurrency limit as a **400** (looks like a deterministic bad-request but is transient). `open_meteo_client.py` now: (1) raises `OpenMeteoThrottled(OpenMeteoError)` for 429 + the throttle reasons and retries ONLY those (+5xx/transport) via `AsyncRetrying` with exponential backoff+jitter — genuine 4xx (bad params, HRRR out-of-CONUS) stay non-retryable; (2) sleeps a random `0..JITTER_SECONDS` per worker before its first call to spread the herd; (3) hard-bounds each attempt with `asyncio.wait_for(_, REQUEST_TIMEOUT_SECONDS)` so worst-case wall-clock (`JITTER 20 + 4×18 + 3×6 = 110s`, since the two `_get` calls run concurrently under one jitter) stays under the 120s worker timeout. The throttle check is ordered BEFORE the generic error check (the throttle body also has `error:true`). All knobs are env-tunable (`WEATHER_FETCH_JITTER_SECONDS|RETRY_ATTEMPTS|RETRY_WAIT_MAX|TIMEOUT`) and, as of the P-observability fix, ARE wired into Terraform on `weather-worker` (`env` map in `terraform/modules/functions/main.tf`: JITTER=20, TIMEOUT=18, RETRY_WAIT_MAX=6, RETRY_ATTEMPTS=4 — retuned for the 39-mountain herd; code defaults now match). Above ~90 mountains, raise weather-worker `max_instances` (currently 100) or the Cloud Run 429s start counting against the DLQ's 5-attempt budget. Data self-heals each hourly run regardless.
- **Live-only e2e specs must self-gate** with `test.skip(!process.env.PLAYWRIGHT_BASE_URL, "live-only")` so the route-mocked `npm run test:e2e` doesn't run them without a fixture (see `tests/e2e/whitney.spec.ts`).
- **Every `pipeline_error` log MUST carry `errorClass` ("transient"|"actionable") or it will NOT page.** The `pipeline-worker-errors` alert filters on `jsonPayload.errorClass="actionable"` (transient self-healing upstream failures — Open-Meteo/NWAC/NRCS/CDSE timeouts — deliberately do NOT email; they're covered by the `*-pipeline-stale` 3h-no-success absence alerts, and hard-down cases still hit the DLQ alert). The weather worker classifies precisely via its OpenMeteo exception taxonomy (`OpenMeteoUnavailable`/`OpenMeteoThrottled` → transient, bad-params/unexpected → actionable); the other workers use the shared `obs.classify_exception(exc)` heuristic (timeout/connection/transport type-names → transient, else actionable). A NEW worker (or a new `pipeline_error` call site) that omits `errorClass` silently drops out of the alert — always pass it. Root cause of the 2026-07-03 email storm: a raw `asyncio.TimeoutError` was wrapped as `OpenMeteoError(str(exc))` = `OpenMeteoError("")`, logging blank `error=""`; the fix wraps non-`OpenMeteoError` primary failures as `OpenMeteoUnavailable(str(exc) or repr(exc))`.
- **weather_worker/main.py error-log ordering:** `handle_message` wraps `_handle(mountain_id)` with `except omc.OpenMeteoError: raise` (already logged with its class inside `_handle`) BEFORE the `except Exception` actionable catch-all — reversing them double-logs / misclassifies. Only the fetch is wrapped in snotel/nwac/satellite, so a post-fetch bug there is DLQ-only (not on the fast alert) — a known asymmetry, follow-up if it matters.
